# ADR-0001: Dependency Injection

## Статус

Accepted

---

## Контекст

Проект построен на основе Clean Architecture с пятью слоями: Presentation, Application, Domain, Infrastructure, Composition. Каждый слой имеет строгие правила зависимостей: зависимости направлены к Domain, внешние слои зависят от внутренних через порты (интерфейсы).

В текущем состоянии проект не имеет ни одного реализованного Use Case, порта или инфраструктурного компонента. Composition Layer частично реализован (создание FastAPI-приложения, загрузка ключей, конфигурация), но не содержит механизма создания Use Cases и связывания их с реализациями портов.

По мере развития проекта количество компонентов будет расти: Use Cases (каждый может зависеть от нескольких портов), репозитории (каждый зависит от соединения с БД), сервисы инфраструктуры (зависят от внешних клиентов: Redis, MinIO и т.д.), схемы настройки соединений (пулы подключений, жизненный цикл).

Без явно определённого подхода к DI возникает риск:

- произвольного создания объектов внутри Use Cases;
- нарушения правила "явность важнее магии";
- смешения ответственности между слоями;
- усложнения тестирования.

Необходимо определить подход к созданию объектов, организации Composition Root и внедрению зависимостей.

---

## Требования

1. **Соответствие Clean Architecture** — Composition Layer является единственным местом сборки зависимостей.
2. **Независимость домена** — Domain Layer не должен знать о механизме DI.
3. **Явность важнее магии** — механизм связывания должен быть прозрачным и легко читаемым.
4. **Тестируемость** — возможность замены реализаций портов при тестировании без изменения кода Use Cases.
5. **Сопровождаемость** — при добавлении нового компонента разработчик должен понимать, где и как зарегистрировать его зависимости.
6. **Безопасность** — DI-механизм не должен выполнять неявный код, влияющий на безопасность.
7. **Производительность** — создание объектов не должно быть узким местом; допустим компромисс в пользу явности.

---

## Рассматриваемые варианты

### Вариант 1: Pure DI (ручное связывание в Composition Root)

#### Принцип работы

Composition Root содержит фабричные функции или классы, которые вручную создают все компоненты и связывают их. Например, функция `build_create_user_use_case()` создаёт экземпляр `CreateUserUseCase`, передавая в него реализации портов.

```python
# Пример (не реализация)
def build_create_user_use_case(
    session_factory: Callable[[], AsyncConnection],
    password_hasher: PasswordHasher,
    token_service: TokenService,
    user_repository: UserRepository,
) -> CreateUserUseCase:
    return CreateUserUseCase(
        password_hasher=password_hasher,
        token_service=token_service,
        user_repository=user_repository,
        unit_of_work=SqlAlchemyUnitOfWork(session_factory=session_factory),
    )
```

Зависимости, общие для нескольких компонентов (например, соединение с БД), создаются один раз и передаются во все фабрики.

#### Преимущества

- **Максимальная явность** — все зависимости видны в одном месте, нет магии.
- **Нулевая зависимость от библиотек** — не требуется adding external DI frameworks (соблюдение AGENTS.md, п. 3: новые зависимости требуют согласования).
- **Полный контроль** — разработчик явно управляет временем жизни каждого объекта.
- **Простая отладка** — стек вызовов не содержит фреймворк-специфичных обёрток.
- **Лёгкая замена при тестировании** — для тестов пишется отдельная фабрика с мок-реализациями.

#### Недостатки

- **Бойлерплейт** — с ростом числа компонентов растёт объём кода в Composition Root.
- **Ручное управление порядком** — при появлении циклических зависимостей (которых в хорошо спроектированной архитектуре быть не должно) потребуется рефакторинг.
- **Отсутствие автоматического разрешения** — добавление нового параметра в конструктор Use Case требует ручного обновления фабрики.
- **Риск дублирования** — если несколько фабрик создают похожие цепочки зависимостей, может возникнуть дублирование кода.

#### Ограничения

- Требует дисциплины: все создания объектов должны быть только в Composition Layer.
- Не решает проблему управления временем жизни (transient/scoped/singleton) — это реализуется вручную.

#### Влияние на архитектуру

Укрепляет роль Composition Layer как единого места сборки. Не противоречит Clean Architecture.

#### Влияние на тестируемость

Позитивное: для каждого теста можно написать фабрику, создающую ровно те зависимости, которые нужны. Нет скрытых зависимостей.

#### Влияние на сопровождаемость

При большом количестве компонентов (50+ Use Cases) ручное связывание становится громоздким, но остаётся читаемым.

#### Влияние на производительность

Минимальное — нет оверхеда DI-контейнера.

#### Влияние на безопасность

Позитивное: нет неявного вызова кода. Разработчик явно видит, какие объекты создаются и с какими параметрами.

### Вариант 2: DI-контейнер (библиотека, например, `dependency-injector` или `lagom`)

#### Принцип работы

Используется сторонняя библиотека, которая регистрирует типы и их зависимости в контейнере. Контейнер автоматически разрешает (resolve) зависимости при создании объекта. Composition Root содержит конфигурацию контейнера.

```python
# Пример с dependency-injector (не реализация)
container = Container()
container.wire(modules=[...])
```

#### Преимущества

- **Автоматическое разрешение** — добавление новой зависимости в конструктор не требует изменения конфигурации, если зависимость уже зарегистрирована.
- **Управление жизненным циклом** — большинство контейнеров предоставляют transient/singleton/scoped из коробки.
- **Меньше бойлерплейта** — одна регистрация может покрыть все Use Cases, использующие данный порт.
- **Проверка графа зависимостей** — некоторые контейнеры могут обнаружить Missing dependencies на этапе конфигурации.

#### Недостатки

- **Нарушение "явность важнее магии"** — разрешение зависимостей происходит неявно, разработчик должен знать конфигурацию контейнера, чтобы понять, какая реализация будет внедрена.
- **Зависимость от внешней библиотеки** — требуется согласование (AGENTS.md, п. 3).
- **Усложнение отладки** — при ошибке разрешения стек вызовов содержит код контейнера.
- **Риск переиспользования контейнера в тестах** — может привести к связанным тестам.
- **Дополнительная точка отказа** — неправильная конфигурация контейнера может проявиться только в runtime.

#### Ограничения

- Не все контейнеры поддерживают асинхронное создание объектов.
- Некоторые библиотеки (например, `dependency-injector`) имеют сложный API.
- Не все контейнеры хорошо интегрируются с FastAPI (может потребоваться адаптация).

#### Влияние на архитектуру

Снижает барьер для нарушения Clean Architecture, так как контейнер может быть использован для доступа к инфраструктурным объектам из любого слоя (при отсутствии дисциплины).

#### Влияние на тестируемость

Зависит от подхода: при правильном использовании (фабрики для тестов) — нейтрально. При использовании контейнера в тестах — негативно (тесты становятся зависимыми от конфигурации контейнера).

#### Влияние на сопровождаемость

Для новых разработчиков потребуется изучение API контейнера. При большом количестве зарегистрированных компонентов конфигурация может стать сложной.

#### Влияние на производительность

Незначительный оверхед на разрешение зависимостей. Для приложений среднего размера (сотни компонентов) — не критично.

#### Влияние на безопасность

Зависит от библиотеки: потенциальный риск при использовании непроверенной или неподдерживаемой библиотеки.

### Вариант 3: FastAPI `Depends` на уровне Application Layer

#### Принцип работы

Использовать встроенный механизм FastAPI `Depends` для внедрения зависимостей не только в роутеры, но и на уровень Application Layer, передавая Use Cases как callable-зависимости.

```python
# Пример (не реализация)
@router.post("/users")
async def create_user(
    use_case: CreateUserUseCase = Depends(get_create_user_use_case),
    body: CreateUserRequest = Body(...),
) -> StandardResponse:
    ...
```

Где `get_create_user_use_case` — функция, создающая Use Case и все его зависимости.

#### Преимущества

- **Без дополнительных библиотек** — используется встроенный механизм FastAPI.
- **Тесная интеграция с Presentation Layer** — единый механизм DI во всём приложении.
- **Простота** — разработчики, знакомые с FastAPI, уже знают `Depends`.

#### Недостатки

- **Нарушение Clean Architecture** — Use Cases начинают зависеть от FastAPI (через механизм `Depends`), что прямо запрещено ARCHITECTURE.md.
- **Смешение ответственности** — Presentation Layer должен знать о создании Application и Infrastructure объектов.
- **Сложность тестирования** — тестирование Use Case требует запуска FastAPI или создания Depends-совместимых объектов.
- **Усложнение при росте** — при большом количестве Use Cases функции-провайдеры становятся громоздкими.
- **Отсутствие управления жизненным циклом** — `Depends` не предоставляет scoped/singleton из коробки (только caching через `lru_cache`).

#### Ограничения

- `Depends` работает только в контексте FastAPI (запрос-ответ).
- Не поддерживает асинхронное создание вне FastAPI-цикла.

#### Влияние на архитектуру

Фундаментальное нарушение Clean Architecture: слой Presentation начинает контролировать создание объектов Application и Infrastructure, что противоречит правилу зависимостей.

#### Влияние на тестируемость

Негативное: Use Cases, использующие `Depends`, не могут быть протестированы без FastAPI.

#### Влияние на сопровождаемость

Нарушает правило независимости слоёв: изменение в Application Layer может потребовать изменения в Presentation Layer.

#### Влияние на производительность

Незначительный оверхед на кэширование результатов `Depends`.

#### Влияние на безопасность

Негативное: механизм `Depends` выполняется в контексте HTTP-запроса, что может привести к непреднамеренному доступу к зависимостям.

| Критерий | Pure DI | DI-контейнер | FastAPI Depends |

| Соответствие Clean Architecture | Полное | Возможно при дисциплине | Нарушает |
| Явность (отсутствие магии) | Высокая | Низкая (неявное разрешение) | Средняя |
| Тестируемость | Высокая | Средняя | Низкая (зависимость от FastAPI) |
| Бойлерплейт | Высокий | Низкий | Средний |
| Зависимость от библиотек | Нет | Да | Нет (встроенный механизм) |
| Управление жизненным циклом | Ручное | Автоматическое | Ограниченное |
| Сложность отладки | Низкая | Высокая | Средняя |
| Порог входа | Низкий | Средний/Высокий | Низкий (для знакомых с FastAPI) |
| Производительность | Нативная | Незначительный оверхед | Незначительный оверхед |
| Безопасность | Прозрачная | Зависит от библиотеки | Средняя |

## Рекомендация

Рекомендуется использовать **Pure DI (ручное связывание в Composition Root)**.

Обоснование:

1. **Соответствие архитектурным принципам** — Pure DI полностью соответствует Clean Architecture и правилу "явность важнее магии". Все зависимости создаются явно в Composition Layer, который для этого и предназначен.
2. **Отсутствие зависимости от библиотек** — не требуется согласование и добавление новой зависимости (AGENTS.md, п. 3).
3. **Максимальная тестируемость** — для тестов можно написать отдельную фабрику, не зависящую ни от контейнера, ни от FastAPI.
4. **Прозрачность** — разработчик всегда видит, какой объект и с какими зависимостями создаётся.

При росте проекта (50+ Use Cases) Pure DI может приводить к увеличению бойлерплейта. В этом случае может быть рассмотрено внедрение DI-контейнера, но это решение должно приниматься отдельно на основе конкретной оценки сложности.

Использование FastAPI `Depends` на уровне Application Layer **не рекомендуется**, так как это прямо нарушает Clean Architecture и независимость слоёв.

---

## Решение

Принят **Вариант 1: Pure DI (ручное связывание в Composition Root)**.

Использование DI-контейнера и FastAPI `Depends` на уровне Application Layer отклонены по причинам, изложенным в разделе «Рекомендация».

Пересмотр решения в пользу DI-контейнера допускается в будущем, но должен оформляться отдельным ADR с конкретной оценкой сложности Composition Root на тот момент, а не приниматься по умолчанию при достижении числа Use Cases.

## Ответы на вопросы разработчику

### 1. Группировка фабрик в Composition Layer

Принят подход **одна функция на каждый Use Case**.

Переиспользуемые цепочки зависимостей (например, создание репозитория, unit of work) выносятся в отдельные под-фабрики, которые вызываются из фабрик Use Case. Группировка по DomainContext или единый класс-сборщик (`ApplicationFactory`) не используются, так как это приближает Composition Layer к поведению DI-контейнера (регистрация/резолвинг через класс), что противоречит принципу «явность важнее магии».

### 2. `lru_cache` для singleton-компонентов

Использование `lru_cache` для кэширования дорогих ресурсов (пул соединений с БД, HTTP-клиенты, аналогично уже применяемым `get_settings`, `get_signature_keys`) **допускается**.

Это решение фиксируется как осознанное и документированное исключение из чистого Pure DI: `lru_cache` вносит элемент неявного кэширования, поэтому каждое такое место обязано:

- быть явно задокументировано (NumPy Doc, блок Notes) с указанием, что объект кэшируется и почему;
- ограничиваться исключительно дорогими ресурсами уровня Infrastructure/Composition (пулы соединений, клиенты внешних сервисов), а не применяться к Use Cases или доменным объектам.

### 3. Управление временем жизни

На текущем этапе проекта принят **базовый подход**, без формализации полной схемы transient/scoped/singleton:

- дорогие ресурсы (пул БД, HTTP-клиенты) — один экземпляр на всё приложение (singleton через `lru_cache`, см. п. 2);
- Use Cases — новый экземпляр на каждый вызов (transient).

Формализация схемы времени жизни (включая scoped) откладывается до появления реального кандидата на такой уровень жизненного цикла и будет оформлена отдельным ADR при необходимости.
