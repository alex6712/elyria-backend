# ADR-0003: Authentication Session Model

## Статус

Proposed

---

## Контекст

В проекте уже реализована миграция базы данных, создающая таблицу `user_sessions` со следующей схемой:

- `id` (UUID, PK);
- `user_id` (FK → users, ON DELETE CASCADE);
- `refresh_token_hash` (String 128, HMAC-SHA256 хеш refresh token);
- `expires_at` (DateTime);
- `last_used_at` (DateTime);
- `created_at` (DateTime).

Также в конфигурации (`settings.py`) определены параметры, связанные с аутентификацией:

- `JWS_ALGORITHM = "Ed25519"` (алгоритм цифровой подписи);
- `ACCESS_TOKEN_LIFETIME_MINUTES = 15` (время жизни access token);
- `REFRESH_TOKEN_LIFETIME_DAYS = 5` (время жизни refresh token);
- `HMAC_SECRET_KEY` (секретный ключ для HMAC-хеширования refresh token);
- `REFRESH_TOKEN_COOKIE_NAME = "elyria_rt"`;
- `AUTH_COOKIE_PATH`, `AUTH_COOKIE_SECURE`, `AUTH_COOKIE_SAMESITE`, `AUTH_COOKIE_DOMAIN`.

Ключи для подписи JWT (Ed25519) уже сгенерированы и загружаются (`signature_keys.py`).

На данный момент не реализован ни один сервис аутентификации: нет генерации токенов, нет валидации, нет механизма refresh, нет logout.

Необходимо определить модель хранения пользовательских сессий с учётом существующей схемы БД, жизненный цикл refresh token, стратегию logout и возможность расширения до полноценного управления пользовательскими устройствами.

---

## Требования

1. **Независимость домена** — модель сессии не должна зависеть от HTTP, JWT или конкретных технологий.
2. **Безопасность важнее производительности** — при конфликте требований безопасность имеет приоритет.
3. **Явность важнее магии** — механизм аутентификации должен быть прозрачным.
4. **Соответствие существующей схеме БД** — таблица `user_sessions` уже существует, её изменение требует согласования (AGENTS.md, п. 4).
5. **Поддержка refresh-сессии** — refresh token должен иметь возможность ротации (rotation).
6. **Возможность logout** — однозначное завершение сессии (одной или всех).
7. **Расширяемость до управления устройствами** — архитектура должна позволять в будущем добавить информацию об устройстве (User-Agent, IP, тип устройства, доверенные устройства).
8. **Тестируемость** — сервисы аутентификации должны быть тестируемы без реальных ключей или БД.

---

## Рассматриваемые варианты

### Вариант 1: Полностью stateless (JWT без хранения сессий)

#### Принцип работы

Access token и refresh token являются самодостаточными JWT, подписанными Ed25519. Refresh token содержит все данные, необходимые для валидации (user_id, expires_at, и т.д.). Сервер не хранит refresh token в БД.

Таблица `user_sessions` не используется для аутентификации (может быть удалена или оставлена для аудита).

Logout реализуется через чёрный список (например, Redis) на время жизни токена. Access token имеет короткое время жизни (15 минут), refresh token — 5 дней.

#### Преимущества

- **Максимальная производительность** — отсутствие запросов к БД при валидации токена.
- **Простота масштабирования** — любой сервер может проверить токен без общего хранилища сессий.
- **Меньше зависимостей** — не требуется БД для сессий (сохраняется только Redis для чёрного списка).

#### Недостатки

- **Невозможность принудительного завершения сессии** — если не использовать чёрный список, токен действителен до истечения срока.
- **Чёрный список (Redis) как компромисс** — добавляет ещё одну точку отказа и complexity.
- **Отсутствие управления устройствами** — нет информации о том, с каких устройств пользователь залогинен.
- **Невозможность обнаружения кражи токена** — refresh token rotation (статистически) не хранится, нельзя определить, что токен использован повторно.
- **Противоречие существующей схеме** — таблица `user_sessions` остаётся неиспользованной.

#### Ограничения

- Полная отвязка от существующей таблицы `user_sessions` (требует согласования).
- Чёрный список должен иметь TTL, равный времени жизни токена, что увеличивает нагрузку на Redis.

#### Влияние на архитектуру

Упрощает инфраструктуру (меньше таблиц), но требует Redis (уже настроен). Не противоречит Clean Architecture, так как технология JWT и Redis находятся в Infrastructure.

#### Влияние на тестируемость

Высокая: токены генерируются и проверяются без БД. Чёрный список легко мокируется.

#### Влияние на сопровождаемость

Средняя: чёрный список требует мониторинга, но в остальном код прост.

#### Влияние на производительность

Наилучшая среди вариантов: валидация JWT — O(1), без БД.

#### Влияние на безопасность

Средняя:

- Refresh token rotation затруднён (не с чем сравнить при повторном использовании).
- Аннулирование всех сессий пользователя требует чёрного списка.
- Кража refresh token не обнаруживается на уровне сессии.
- Чёрный список может быть очищен по TTL раньше, чем истечёт токен (race condition).

### Вариант 2: Сессии в БД с refresh token rotation (с использованием `user_sessions`)

#### Принцип работы

Access token — короткоживущий JWT (15 минут), подписанный Ed25519, содержащий `user_id`, `session_id`, `exp`.

Refresh token — длинный случайный токен (не JWT), хеш которого (HMAC-SHA256) хранится в таблице `user_sessions`. Refresh token используется только для получения новой пары токенов.

При каждом refresh:

1. Текущий refresh token проверяется на наличие в БД (по хешу);
2. Если найден — создаётся новый refresh token, старый удаляется или помечается как использованный (rotation);
3. Если не найден (повторное использование украденного токена) — все сессии пользователя аннулируются (предполагается кража токена).

Logout:

- Удаляется запись из `user_sessions` (конкретная сессия);
- Либо удаляются все записи пользователя (logout all devices).

#### Преимущества

- **Соответствие существующей схеме** — таблица `user_sessions` используется по назначению, не требуется изменений БД.
- **Обнаружение кражи токена** — повторное использование refresh token приводит к аннулированию всех сессий.
- **Возможность управления сессиями** — список сессий пользователя доступен через запрос к БД.
- **Расширяемость до устройств** — в таблицу можно добавить поля `user_agent`, `ip_address`, `device_name`, `is_trusted`.
- **Принудительный logout** — удаление/изменение записи в БД немедленно завершает сессию.

#### Недостатки

- **Запрос к БД при каждом refresh** — дополнительная нагрузка (но refresh происходит раз в 15 минут, что не критично).
- **Сложность реализации** — требуется реализовать механизм rotation с защитой от race condition (concurrent refresh requests).
- **Хранение хешей** — не сам токен, а его HMAC-хеш. Это плюс с точки зрения безопасности, но минус с точки зрения сложности.
- **Зависимость от БД** — если БД недоступна, refresh невозможен (но это приемлемо; access token продолжает работать 15 минут).

#### Ограничения

- Refresh token должен быть достаточно длинным и криптостойким (например, 32 байта случайных данных, закодированных в base64).
- Race condition при rotation (два параллельных refresh-запроса с одним токеном) — требуется защита.

#### Влияние на архитектуру

Соответствует Clean Architecture: порт `SessionRepository` (или `TokenRepository`) в Application Layer, реализация в Infrastructure (SQLAlchemy Core). Домен не знает о refresh token механизме.

#### Влияние на тестируемость

Высокая: сессионный репозиторий заменяется моком.

#### Влияние на сопровождаемость

Высокая: модель данных явно отражает бизнес-логику (сессии пользователя).

#### Влияние на производительность

Средняя: дополнительный запрос к БД при каждом refresh. Для среднестатистического приложения (тысячи пользователей) — некритично.

#### Влияние на безопасность

Высокая:

- Refresh token rotation с аннулированием при повторном использовании.
- Хранение HMAC-хеша, а не токена в БД (компрометация БД не раскрывает токены).
- Возможность принудительного завершения всех сессий пользователя.
- Возможность обнаружения кражи refresh token.

### Вариант 3: Гибрид (Redis + БД)

#### Принцип работы

Access token — JWT (15 минут). Refresh token — случайный токен, хеш хранится в Redis (быстрый доступ). Таблица `user_sessions` используется для долговременного хранения (история сессий, устройства).

При каждом refresh:

1. Проверка хеша в Redis (O(1));
2. Создание нового refresh token (новый хеш в Redis);
3. Обновление `last_used_at` в `user_sessions` асинхронно (или через фоновую задачу).

Logout:

- Удаление ключа из Redis + обновление статуса в `user_sessions`.

#### Преимущества

- **Быстрый доступ** — Redis обеспечивает скорость проверки refresh token.
- **Наличие истории в БД** — таблица `user_sessions` содержит полную историю.
- **Меньше нагрузки на БД** — refresh token rotation не создаёт транзакций в PostgreSQL.

#### Недостатки

- **Сложность** — два хранилища (Redis + PostgreSQL) с необходимостью синхронизации.
- **Риск рассинхронизации** — если Redis потеряет данные (сброс, очистка), сессии становятся недействительными, хотя БД содержит записи.
- **Усложнение тестирования** — требуется мокировать и Redis, и репозиторий.
- **Избыточность** — для текущего масштаба проекта два хранилища не оправданы.

#### Ограничения

- Redis должен быть настроен на персистентность, если сессии должны переживать перезапуск.
- При потере Redis пользователям потребуется заново логиниться.

#### Влияние на архитектуру

Допустимо в рамках Clean Architecture, но добавляет сложность в Infrastructure Layer.

#### Влияние на тестируемость

Средняя: требуется мокировать два сервиса.

#### Влияние на сопровождаемость

Сложнее, чем каждый из вариантов по отдельности: два хранилища, два набора мониторинга.

#### Влияние на производительность

Высокая: refresh token проверяется через Redis (in-memory).

#### Влияние на безопасность

Хорошая: хеш в Redis + история в БД. Но уязвимость к потере Redis (потеря активных сессий).

## Сравнительная таблица

| Критерий | Stateless (JWT) | Сессии в БД | Гибрид (Redis + БД) |
|---|---|---|---|
| Соответствие существующей схеме | Нет | Да | Частично |
| Безопасность (кража токена) | Низкая | Высокая | Высокая |

| Сложность реализации | Средняя | Средняя | Высокая |

| Расширяемость до устройств | Низкая | Высокая | Средняя |

| Обнаружение многократного использования токена | Нет | Да | Да |

## Рекомендация

Рекомендуется использовать **Вариант 2: Сессии в БД с refresh token rotation**.

Обоснование:

1. **Соответствие существующей схеме** — таблица `user_sessions` уже создана; использование её по назначению не требует изменений БД или согласования миграций.
2. **Безопасность** — механизм rotation с аннулированием при повторном использовании является наиболее безопасным среди рассмотренных вариантов, что соответствует принципу "безопасность важнее производительности".
3. **Явность** — модель данных (таблица `user_sessions`) явно отражает состояние сессий; любой разработчик может запросом к БД увидеть активные сессии пользователя.
4. **Расширяемость** — добавление полей `user_agent`, `ip_address`, `device_name` в существующую таблицу не нарушает архитектуру и даёт возможность управления устройствами в будущем.
5. **Достаточная производительность** — refresh token проверяется раз в 15 минут; для проекта на данном этапе этого более чем достаточно. При росте нагрузки может быть рассмотрена миграция на гибридную схему (Redis + БД).

Гибридная схема (Вариант 3) избыточна на данном этапе. Stateless (Вариант 1) жертвует безопасностью и противоречит существующей схеме БД.

## Зависимости

Данный ADR зависит от решений, принятых в:

- ADR-0001 (Dependency Injection) — сервисы аутентификации (TokenService, SessionRepository) будут создаваться через Composition Root.
- ADR-0002 (Transaction Management) — операция создания пользователя и создания сессии должны быть атомарными в рамках одного Use Case.

## Вопросы разработчику

1. Должен ли refresh token rotation аннулировать все сессии пользователя при обнаружении повторного использования токена (детектирование кражи) или только текущую сессию?
2. Какой механизм защиты от race condition при параллельных refresh-запросах предпочтителен: блокировка на уровне БД (SELECT FOR UPDATE) или optimistic locking с использованием `last_used_at`?
3. Следует ли добавить в таблицу `user_sessions` поля для расширения до управления устройствами (например, `user_agent`, `ip_address`, `device_name`, `is_trusted`) на данном этапе или отложить до появления соответствующей функциональности?
4. Какой алгоритм генерации refresh token использовать: `os.urandom(32)` с base64-кодированием или UUIDv4? При использовании `os.urandom` токен криптостойкий, но неудобен для отладки. UUIDv4 удобнее, но имеет меньшую энтропию (122 бита против 256).
5. Как часто разрешается использовать один и тот же refresh token до его ротации? Должен ли refresh token быть одноразовым (использован → удалён) или допускается повторное использование в течение короткого окна (например, 1 секунда) для компенсации race condition?
