# ADR-0002: Transaction Management

## Статус

Accepted

---

## Контекст

В проекте используется SQLAlchemy Core (без ORM). `ARCHITECTURE.md` в явном виде устанавливает:

- репозитории не выполняют commit транзакции (п. "Репозитории");
- архитектура управления транзакциями будет определена отдельным ADR (п. "Работа с базой данных").

На данный момент в проекте нет ни одного репозитория, ни одного Use Case и ни одной реализации Unit of Work.

При выполнении Use Case может потребоваться выполнение нескольких операций с разными репозиториями (например, создание пользователя и создание сессии). Эти операции должны выполняться атомарно: при ошибке на любом шаге все изменения должны быть отменены.

Кроме того, в рамках одного Use Case могут выполняться операции, которые не зависят друг от друга (например, запись в кэш, загрузка файла в MinIO, отправка уведомления). Нетранзакционные операции не должны блокироваться транзакцией БД.

Дополнительная сложность: проект асинхронный (asyncpg), SQLAlchemy Core предоставляет два уровня работы с транзакциями — на уровне подключения (`connection.begin()`) и на уровне движка (`engine.begin()`).

Необходимо определить:

- кто управляет транзакцией;
- как организован Unit of Work;
- как обрабатываются конкурентные независимые операции внутри одного Use Case;
- как транзакция интегрируется с Composition Root.

---

## Требования

1. **Атомарность в рамках агрегата** — изменения, относящиеся к одному бизнес-действию, должны быть атомарными.
2. **Явное управление транзакциями** — Use Case должен явно указывать начало и завершение транзакции или делегировать это Unit of Work.
3. **Независимость домена** — Domain Layer не должен знать о транзакциях.
4. **Принцип "репозиторий не делает commit"** — соблюдение установленного правила ARCHITECTURE.md.
5. **Конкурентность независимых операций** — операции, не требующие транзакции (логирование, уведомления, запись в кэш), должны иметь возможность выполняться конкурентно с транзакционными операциями.
6. **Тестируемость** — возможность тестирования Use Case с транзакционным поведением без подключения к реальной БД.
7. **Безопасность** — предотвращение частичного применения изменений при сбоях (rollback при исключении).

---

## Рассматриваемые варианты

### Вариант 1: Unit of Work с явным управлением

#### Принцип работы

Вводится абстракция `UnitOfWork`, которая инкапсулирует транзакцию. Use Case получает `UnitOfWork` через порт (интерфейс в Application Layer). Use Case явно вызывает `uow.begin()`, `uow.commit()`, `uow.rollback()`. Репозитории получают подключение от `UnitOfWork`.

```python
# Пример (не реализация)
class UnitOfWork(ABC):
    @abstractmethod
    async def begin(self) -> None: ...
    @abstractmethod
    async def commit(self) -> None: ...
    @abstractmethod
    async def rollback(self) -> None: ...
    @abstractmethod
    def get_connection(self) -> AsyncConnection: ...
```

Use Case:

```python
class CreateUserUseCase:
    def __init__(self, uow: UnitOfWork, user_repo: UserRepository):
        self._uow = uow
        self._user_repo = user_repo

    async def execute(self, command: CreateUserCommand) -> User:
        await self._uow.begin()
        try:
            user = User.create(command)
            await self._user_repo.add(user)
            await self._uow.commit()
            return user
        except Exception:
            await self._uow.rollback()
            raise
```

Репозиторий получает `AsyncConnection` от `UnitOfWork` и не знает о транзакции.

Независимые операции (например, отправка email) выполняются после commit или в отдельном фоновом задании.

#### Преимущества

- **Явность** — Use Case явно управляет транзакцией (try/commit/rollback), что соответствует принципу "явность важнее магии".
- **Чёткая ответственность** — `UnitOfWork` отвечает только за управление транзакцией, репозиторий — только за чтение/запись.
- **Атомарность** — гарантируется через явный commit/rollback.
- **Тестируемость** — `UnitOfWork` заменяется моком; можно проверить, что commit был вызван при успехе и rollback — при ошибке.
- **Безопасность** — явный rollback в `except` блоке гарантирует откат при любом исключении.

#### Недостатки

- **Бойлерплейт** — каждый Use Case содержит try/except/commit/rollback.
- **Риск забыть commit** — разработчик может пропустить вызов commit.
- **Риск забыть rollback** — при необработанном исключении транзакция может остаться открытой (но пул asyncpg обычно закрывает такие соединения при возврате в пул).

#### Ограничения

- Не решает проблему независимых конкурентных операций — они должны выполняться вне `UnitOfWork`.
- Каждый `UnitOfWork` привязан к одному подключению (connection), что ограничивает параллельные операции в рамках одной транзакции.

#### Влияние на архитектуру

Соответствует Clean Architecture: `UnitOfWork` — порт в Application Layer, реализация в Infrastructure Layer. Репозитории не нарушают правило "не делать commit".

#### Влияние на тестируемость

Высокая: `UnitOfWork` легко заменяется моком.

#### Влияние на сопровождаемость

Требует дисциплины: каждый Use Case должен явно управлять транзакцией. При большом количестве Use Cases появляется шаблонный код.

#### Влияние на производительность

Минимальное: явное создание транзакции добавляет один дополнительный вызов.

#### Влияние на безопасность

Позитивное: явный rollback при ошибке предотвращает неконсистентное состояние БД.

### Вариант 2: Контекстный менеджер (Unit of Work как context manager)

#### Принцип работы

`UnitOfWork` реализует протокол асинхронного контекстного менеджера. Use Case оборачивает тело сценария в `async with uow:`. Commit и rollback выполняются автоматически при выходе из контекста.

```python
# Пример (не реализация)
class UnitOfWork:
    async def __aenter__(self):
        self._connection = await self._engine.connect()
        self._transaction = await self._connection.begin()
        return self

    async def __aexit__(self, exc_type, exc_val, exc_tb):
        if exc_type is not None:
            await self._transaction.rollback()
        else:
            await self._transaction.commit()
        await self._connection.close()
```

Use Case:

```python
class CreateUserUseCase:
    def __init__(self, uow_factory: Callable[[], UnitOfWork], user_repo: UserRepository):
        self._uow_factory = uow_factory
        self._user_repo = user_repo

    async def execute(self, command: CreateUserCommand) -> User:
        async with self._uow_factory() as uow:
            user = User.create(command)
            await self._user_repo.add(user)
            return user
```

#### Преимущества

- **Меньше бойлерплейта** — commit/rollback автоматические.
- **Безопасность** — rollback гарантирован при любом исключении (даже если забыт try).
- **Читаемость** — границы транзакции видны визуально через отступ контекста.
- **Явность** — хотя commit автоматический, граница транзакции явно обозначена конструкцией `async with`.

#### Недостатки

- **Меньшая гибкость** — если нужно выполнить частичный commit (что редко встречается), контекстный менеджер усложняется.
- **Риск избыточного объёма** — разработчик может включить в контекст операции, не относящиеся к транзакции.
- **Тестирование** — требуется проверка, что `__aexit__` вызван при исключении.

#### Ограничения

- Работает только для синхронных или асинхронных сценариев (в проекте — асинхронные).
- Как и Вариант 1, не решает проблему независимых конкурентных операций.

#### Влияние на архитектуру

Соответствует Clean Architecture. Аналогично Варианту 1.

#### Влияние на тестируемость

Высокая: можно мокировать фабрику `UnitOfWork`.

#### Влияние на сопровождаемость

Выше, чем у Варианта 1: меньше шаблонного кода, автоматическая обработка ошибок.

#### Влияние на производительность

Аналогично Варианту 1.

#### Влияние на безопасность

Лучше, чем Вариант 1: rollback невозможно забыть, так как он выполняется автоматически.

### Вариант 3: Transactional decorator (декоратор Use Case)

#### Принцип работы

Вводится декоратор `@transactional`, который оборачивает метод `execute` Use Case, автоматически создавая транзакцию.

```python
# Пример (не реализация)
@transactional
async def execute(self, command: CreateUserCommand, uow: UnitOfWork) -> User:
    user = User.create(command)
    await self._user_repo.add(user)
    return user
```

Декоратор перехватывает исключения и выполняет rollback. Commit выполняется при успешном завершении.

#### Преимущества

- **Минимум бойлерплейта** — самая компактная запись среди всех вариантов.
- **Безопасность** — rollback гарантирован.

#### Недостатки

- **Нарушение "явность важнее магии"** — разработчик не видит границ транзакции; факт наличия транзакции скрыт за декоратором.
- **Сложность отладки** — при ошибке разработчику нужно знать, какие декораторы применены.
- **Сложность тестирования** — требуется проверка поведения декоратора, а не только Use Case.
- **Привязка к типу Use Case** — декоратор должен знать, как получить `UnitOfWork` или подключение.
- **Проблема с параметрами** — если декоратор неявно внедряет зависимости (например, `uow`), это нарушает принцип явности.

#### Ограничения

- Декораторы в Python имеют ограничения при работе с асинхронными методами (требуется правильно реализовать поддержку корутин).
- Если в Use Case несколько методов, которые требуют транзакции, декоратор должен применяться к каждому.

#### Влияние на архитектуру

Создаёт риск неявного поведения. Декоратор, работающий с транзакцией, должен быть размещён в Infrastructure Layer (как реализация), но применяться к Use Case в Application Layer, что может размыть границы слоёв.

#### Влияние на тестируемость

Средняя: для тестирования транзакционного поведения нужно знать, что декоратор применён.

#### Влияние на сопровождаемость

Снижает читаемость: при взгляде на Use Case неочевидно, что он транзакционный.

#### Влияние на производительность

Добавляет накладные расходы на вызов декоратора (минимальные).

#### Влияние на безопасность

Хорошая автоматическая обработка rollback, но снижение прозрачности.

### Вариант 4: Конкурентные независимые операции (дополнение к любому варианту)

Это не самостоятельный вариант, а аспект, применимый к любому из предложенных выше.

#### Принцип работы

В рамках одного Use Case могут быть операции, которые не требуют транзакции БД (например, запись в кэш, отправка уведомления, загрузка файла). Они должны выполняться конкурентно, но без привязки к транзакции.

Подходы:

1. **Последовательное выполнение после commit** — сначала транзакционные операции (commit), затем конкурентные. Независимые операции выполняются после завершения транзакции.
2. **Параллельное выполнение вне транзакции** — независимые операции запускаются через `asyncio.gather()` с собственными подключениями (например, Redis), не участвующими в транзакции.
3. **Фоновые задачи** — независимые операции отправляются в фоновую очередь (например, Redis Queue, Task Queue) и выполняются асинхронно вне контекста HTTP-запроса.

#### Преимущества

- **Гибкость** — не транзакционные операции не блокируются транзакцией.
- **Производительность** — конкурентное выполнение независимых операций сокращает общее время выполнения Use Case.

#### Недостатки

- **Сложность** — требуется явное разделение на транзакционные и не транзакционные операции.
- **Риск** — при параллельном выполнении не транзакционных операций до commit может возникнуть состояние, когда данные ещё не сохранены, а уведомление уже отправлено.

#### Влияние на архитектуру

Требует от Use Case явно разделять ответственность: что должно быть атомарно, а что — нет.

## Рекомендация

Рекомендуется использовать **Вариант 2: Unit of Work как асинхронный контекстный менеджер**.

Обоснование:

1. **Явность при автоматизации** — границы транзакции явно обозначены конструкцией `async with`, что соответствует принципу "явность важнее магии", но commit/rollback выполняются автоматически, снижая риск ошибок.
2. **Безопасность** — автоматический rollback при любом исключении исключает сценарии частично применённых изменений.
3. **Соответствие Clean Architecture** — `UnitOfWork` является портом в Application Layer; его реализация с использованием `sqlalchemy.ext.asyncio.AsyncConnection` находится в Infrastructure Layer.
4. **Тестируемость** — фабрика `UnitOfWork` легко заменяется в тестах.
5. **Меньше шаблонного кода** по сравнению с явным управлением.

Вопрос конкурентных независимых операций (Вариант 4) предлагается решать последовательно: сначала транзакционные операции с commit, затем, при необходимости, конкурентное выполнение независимых (не транзакционных) операций с помощью `asyncio.gather()`. Фоновые очереди на данном этапе не требуются.

## Зависимости

Данный ADR зависит от решения, принятого в ADR-0001 (Dependency Injection): `UnitOfWork` должен создаваться через Composition Root и передаваться в Use Case.

---

## Решение

Принят **Вариант 2: Unit of Work как асинхронный контекстный менеджер**.

Использование явного управления (Вариант 1) и transactional-декоратора (Вариант 3) отклонены по причинам, изложенным в разделе «Рекомендация».

Конкурентные независимые операции (Вариант 4) выполняются последовательно после успешного commit транзакционной части; при необходимости — конкурентно между собой через `asyncio.gather()`. Фоновые очереди на данном этапе не вводятся.

При реализации необходимо учитывать, что исключение, возникшее непосредственно на этапе `commit()` внутри `__aexit__`, также должно приводить к явному `rollback()`/корректному закрытию транзакции, а не просто к закрытию соединения — простое закрытие connection не эквивалентно явному откату.

## Ответы на вопросы разработчику

### 1. Scoped vs transient

`UnitOfWork` создаётся **transient** — новый экземпляр на каждый вызов Use Case, через `uow_factory`, привязанную к singleton-пулу соединений (см. ADR-0001, п. 2 ответов). Один HTTP-запрос = один вызов Use Case = один `UnitOfWork` = одна транзакция. Отдельная scoped-сущность на уровне запроса не вводится до появления сценария с несколькими Use Case, разделяющими одну транзакцию.

### 2. Уровень изоляции транзакций

По умолчанию используется **Read Committed** (стандартный уровень PostgreSQL). Уровень изоляции передаётся `UnitOfWork` явным параметром с этим значением по умолчанию — переопределение для конкретного Use Case (например, Serializable) должно быть видно в коде фабрики этого Use Case, а не задаваться неявно внутри реализации Infrastructure Layer.

### 3. Retry-механизм для serializable-аномалий

**Не реализуется на данном этапе.** Решение откладывается: механизм ретраев вводится вместе с первым Use Case, которому реально требуется уровень изоляции Serializable, и оформляется как дополнение к данному ADR либо отдельным ADR — в зависимости от объёма изменений на тот момент.

### 4. Partial success при ошибке независимой операции после commit

Принят подход **(a): допустимый partial success**. Если транзакционная часть Use Case успешно закоммичена, а последующая независимая (не транзакционная) операция завершилась ошибкой — это не считается ошибкой бизнес-сценария и не приводит к попытке компенсации или отката уже закоммиченных данных. Ошибка независимой операции обязательна к логированию с достаточным контекстом (какой Use Case, какая операция, какие идентификаторы сущностей) для последующего ручного или автоматического восстановления при необходимости.

Компенсирующие действия и повторные попытки для конкретных независимых операций могут вводиться отдельно, по мере появления таких требований у конкретных Use Case, но не как общий механизм на уровне UnitOfWork.

### 5. `connection.begin()` vs `engine.begin()`

Используется **`engine.begin()`**. `UnitOfWork.__aenter__` открывает соединение и транзакцию через `engine.begin()`, `__aexit__` явно завершает транзакцию (commit/rollback) и закрывает соединение в рамках того же контекстного блока. Управление временем жизни соединения полностью инкапсулировано внутри `UnitOfWork` и не выходит за его границы.
