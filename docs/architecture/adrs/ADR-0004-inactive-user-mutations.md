# ADR-0004: Поведение мутаторов User для неактивных пользователей

## Статус

Accepted

---

## Контекст

Доменная сущность `User` содержит признак `is_active`, определяющий, является ли учётная запись активной. Помимо `activate()` / `deactivate()`, сущность имеет ряд методов изменения состояния: `change_password`, `change_display_name`, `change_avatar`, `change_username`, `rotate_e2ee_credentials`.

На момент проектирования не было определено, должны ли эти методы быть доступны для пользователя с `is_active=False`. Отсутствие явного решения создавало риск того, что разные Use Case независимо друг от друга примут разные негласные допущения о поведении сущности в неактивном состоянии, что привело бы к рассинхронизации инвариантов домена.

В проекте пока отсутствуют use case, требующие изменения состояния заблокированного пользователя (например, восстановление доступа модератором, смена пароля перед реактивацией и т.д.). Требовалось зафиксировать поведение по умолчанию, не блокируя дальнейшую разработку, и предусмотреть путь для пересмотра при появлении такого сценария.

Необходимо определить, разрешены ли методы изменения состояния `User` для неактивных пользователей на уровне домена.

---

## Требования

1. **Независимость домена** — правило должно быть реализовано исключительно средствами Entity, без обращения к Application/Infrastructure.
2. **Явность важнее магии** — ограничение должно быть выражено явной проверкой и явным исключением, а не побочным эффектом инфраструктуры (например, отказом на уровне БД или HTTP).
3. **Безопасность важнее производительности** — при отсутствии явного use case приоритет отдаётся более строгому инварианту, снижающему поверхность атаки через заблокированную, но всё ещё изменяемую учётную запись.
4. **Документация является частью реализации** — принятое правило должно быть отражено в docstring сущности и оформлено ADR.
5. **Архитектурные решения принимает разработчик** — при наличии нескольких допустимых вариантов агент не выбирает поведение самостоятельно.
6. **Расширяемость** — правило должно допускать последующий пересмотр (частичное разрешение операций) без разрушающих изменений в сигнатурах методов.

---

## Рассматриваемые варианты

### Вариант A: Полный запрет всех мутаций для неактивных пользователей

#### Принцип работы

Все методы изменения состояния (`change_password`, `change_display_name`, `change_avatar`, `change_username`, `rotate_e2ee_credentials`), кроме `activate()`, проверяют `is_active` в начале выполнения и возбуждают `InactiveUserException`, если пользователь деактивирован.

```python
# Пример (не реализация)
def change_password(self, new_password_hash: str) -> None:
    self._ensure_active()
    self.password_hash = new_password_hash
    self._touch()
```

#### Преимущества

- **Максимально строгий инвариант** — заблокированный пользователь гарантированно неизменяем, что минимизирует риск обхода блокировки через косвенные операции.
- **Единая точка проверки** — `_ensure_active()` переиспользуется во всех мутаторах, что соответствует принципу "явность важнее магии": правило видно и читается в каждом методе.
- **Простота обоснования на аудите безопасности** — единственное состояние, в котором объект-сущность допускает изменение, это `is_active=True`; проверка тривиальна.
- **Согласуется с отсутствием use case** — на данный момент нет ни одного сценария, где изменение состояния заблокированного пользователя было бы необходимо, поэтому запрет не создаёт искусственных ограничений для существующей функциональности.

#### Недостатки

- **Потенциальная избыточная строгость** — при появлении легитимного сценария (например, смена пароля для последующей реактивации без обращения к администратору) потребуется точечно ослаблять инвариант.
- **Единообразие может скрывать разную природу операций** — `change_avatar` и `rotate_e2ee_credentials` содержательно различаются по чувствительности, но при варианте A трактуются одинаково.

#### Ограничения

- Требует, чтобы `InactiveUserException` был определён и доступен в Domain Layer.
- Любое расширение (частичное разрешение) потребует пересмотра ADR, а не тихого изменения кода.

#### Влияние на архитектуру

Полностью соответствует Clean Architecture: правило реализовано внутри Entity, не затрагивает Application/Infrastructure/Presentation. Presentation Layer лишь обязан будет замапить `InactiveUserException` на HTTP-код (вне области текущего ADR).

#### Влияние на тестируемость

Позитивное: поведение детерминировано и одинаково для всех мутаторов, что позволяет покрыть его одним параметризованным набором тестов (по одному вызову на метод + проверка идемпотентности `activate`/`deactivate`).

#### Влияние на сопровождаемость

Высокая предсказуемость: разработчик, добавляющий новый метод изменения состояния, обязан либо вызвать `_ensure_active()`, либо явно задокументировать исключение из правила — рассинхронизации не возникает.

#### Влияние на производительность

Незначительное — одна проверка булевого поля на вызов метода.

#### Влияние на безопасность

Позитивное: минимизирует поверхность атаки — скомпрометированная, но деактивированная учётная запись не может быть изменена ни при каких условиях на уровне домена.

---

### Вариант B: Полное разрешение мутаций, `is_active` влияет только на аутентификацию

#### Принцип работы

Методы изменения состояния не проверяют `is_active`. Флаг используется исключительно в Application Layer (например, при попытке входа или выполнении операций от имени пользователя), где выполняется отдельная проверка перед вызовом Use Case.

```python
# Пример (не реализация)
def change_password(self, new_password_hash: str) -> None:
    self.password_hash = new_password_hash
    self._touch()
```

#### Преимущества

- **Минимальный бойлерплейт в Entity** — отсутствие повторяющихся проверок в каждом методе.
- **Гибкость** — любые сценарии изменения состояния заблокированного пользователя (администратором, самим пользователем при восстановлении) реализуются без изменения домена.

#### Недостатки

- **Нарушает принцип "безопасность важнее производительности"** в трактовке приоритета строгости при отсутствии явного требования — инвариант отсутствует там, где мог бы быть.
- **Размывает ответственность Entity** — правило "State the entity в неизменяемое состояние при is_active=False" отсутствует внутри самой сущности, хотя относится исключительно к её собственному состоянию (что по ARCHITECTURE.md — прямая зона ответственности Entity, а не Application Service).
- **Риск рассинхронизации** — при появлении нескольких Use Case, работающих с `User`, каждый обязан самостоятельно помнить о необходимости проверки `is_active`, что при отсутствии централизованной проверки создаёт риск, что один из сценариев её упустит.

#### Ограничения

- Требует дисциплины на уровне каждого Use Case, использующего мутаторы `User`.

#### Влияние на архитектуру

Формально не нарушает Clean Architecture (проверка остаётся в Application Layer, домен не зависит от внешних слоёв), но противоречит правилу "Entity отвечает исключительно за собственное состояние" — фактическая проверка инварианта, относящегося к состоянию самой сущности, вынесена наружу.

#### Влияние на тестируемость

Негативное: для полноты покрытия пришлось бы тестировать проверку `is_active` в каждом Use Case отдельно, вместо одного набора тестов на уровне Entity.

#### Влияние на сопровождаемость

Более низкая: при добавлении нового Use Case разработчик должен помнить о необходимости ручной проверки `is_active`, что не гарантировано на уровне типов или архитектуры.

#### Влияние на производительность

Нейтральное.

#### Влияние на безопасность

Негативное относительно Варианта A: при ошибке в одном из Use Case заблокированный пользователь может быть изменён в обход намерения блокировки.

### Вариант C: Дифференцированный запрет (по методам)

#### Принцип работы

Часть методов (например, `change_password` — для последующего восстановления доступа) разрешена для неактивных пользователей, остальные — запрещены.

```python
# Пример (не реализация)
def change_password(self, new_password_hash: str) -> None:
    # разрешено даже для is_active=False
    self.password_hash = new_password_hash
    self._touch()

def change_username(self, new_username: Username) -> None:
    self._ensure_active()
    ...
```

#### Преимущества

- **Максимальное соответствие потенциальным будущим сценариям** — заранее закладывает поддержку восстановления доступа без повторного пересмотра инварианта.

#### Недостатки

- **Требует конкретного use case уже сейчас** — на момент принятия решения ни один такой сценарий не сформулирован; разделение методов на "разрешённые"/"запрещённые" было бы умозрительным.
- **Нарушает принцип "архитектурные решения принимает разработчик"** — потребовало бы от агента самостоятельно определить, какие именно операции считать безопасными для заблокированного пользователя, при отсутствии на это явного указания.
- **Усложняет аудит** — правило становится не единым инвариантом, а набором частных исключений, что противоречит "явность важнее магии": для понимания поведения сущности потребуется прочитать каждый метод отдельно.

#### Ограничения

- Список "разрешённых" методов пришлось бы пересматривать при каждом изменении набора мутаторов.

#### Влияние на архитектуру

Не нарушает Clean Architecture, но создаёт неоднородный инвариант внутри одной Entity.

#### Влияние на тестируемость

Среднее: требует отдельного теста на каждый метод с фиксацией того, к какой группе он относится, и последующего сопровождения этого списка.

#### Влияние на сопровождаемость

Негативное: при добавлении нового метода изменения состояния разработчику придётся заново решать, к какой группе он относится, без чёткого критерия на текущем этапе проекта.

#### Влияние на производительность

Незначительное.

#### Влияние на безопасность

Смешанное: строгость сохраняется частично, но неоднородность повышает риск ошибочного отнесения нового метода к "разрешённой" группе.

## Рекомендация

Рекомендуется использовать **Вариант A: полный запрет всех мутаций для неактивных пользователей**.

Обоснование:

1. **Соответствие принципу "безопасность важнее производительности"** — при отсутствии конкретного требования приоритет отдаётся более строгому инварианту.
2. **Соответствие принципу "Entity отвечает исключительно за собственное состояние"** — правило реализуется внутри самой сущности, без делегирования проверки в Application Layer.
3. **Отсутствие обосновывающего use case** — ни один из рассматриваемых сценариев частичного разрешения (Вариант C) не сформулирован как реальная потребность на данном этапе; вводить дифференциацию заранее означало бы принимать архитектурное решение без достаточных оснований.
4. **Расширяемость** — при появлении конкретного use case (например, восстановление доступа) решение может быть пересмотрено отдельным ADR, точечно ослабляющим инвариант для конкретных методов, без необходимости менять сигнатуры или структуру сущности.

---

## Решение

Принят **Вариант A: полный запрет всех мутаций для неактивных пользователей**.

Все методы изменения состояния `User`, кроме `activate()`, проверяют `is_active` через `_ensure_active()` и возбуждают `InactiveUserException` при `is_active=False`.

Варианты B и C отклонены по причинам, изложенным в разделе «Рекомендация».

Пересмотр решения в пользу частичного разрешения операций (Вариант C) допускается в будущем при появлении конкретного use case, но должен оформляться отдельным ADR с описанием этого сценария, а не приниматься по умолчанию.
