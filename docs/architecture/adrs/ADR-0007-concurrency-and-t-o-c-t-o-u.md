# ADR-0007: Борьба с TOCTOU при конкурентном доступе к агрегатам

## Статус

Accepted

---

## Контекст

В DDD и Clean Architecture стандартный паттерн изменения агрегата выглядит как "загрузить из репозитория → модифицировать через методы сущности → сохранить". Между шагами "загрузить" и "сохранить" проходит реальное время, в течение которого другой concurrent запрос может изменить ту же сущность.

Классический TOCTOU (Time-of-Check Time-of-Use):

```python
# Загрузили
identity = await repo.get_by_username(username)
# ← в этот момент другой запрос вызвал identity.deactivate()
identity.change_password(new_hash)   # проверка _ensure_active() пройдена
await repo.change_password_hash(...) # перезатрёт изменения конкурента
```

В проекте на момент написания ADR уже существует три use case (RegisterUserUseCase, LoginUseCase, RefreshSessionUseCase), и в двух из них проблема потенциально проявляется. При этом `RefreshSessionUseCase` осознанно отступает от "чистого" DDD-цикла: он не загружает сущность, а выполняет атомарный условный UPDATE в репозитории (`rotate_secret`). Такой разнобой в подходах требует архитектурной фиксации.

Необходимо определить, какой подход к конкурентному доступу должен применяться по умолчанию, и в каких случаях допустимо от него отступать.

---

## Требования

1. **Целостность агрегата** — изменение состояния сущности не должно терять изменения, применённые другим concurrent запросом; итоговое состояние должно соответствовать одному из последовательно выполненных сценариев (linearizability).
2. **Независимость домена** — решение не должно завязывать Domain Layer на механизм блокировок конкретной СУБД.
3. **Явность важнее магии** — механизм конкурентного контроля должен быть явно обозначен в контрактах репозитория, а не спрятан в реализации.
4. **Безопасность важнее производительности** — приоритет консистентности над пропускной способностью.
5. **Совместимость с Unit of Work** — конкурентный контроль не должен требовать выхода за границы существующей транзакции (см. ADR-0002).
6. **Минимизация retry-логики** — use case не должен требовать обязательного retry-цикла; отказ допустим, но должен быть явным и детерминированным.

---

## Рассматриваемые варианты

### Вариант A: Атомарные условные UPDATE в репозиториях (условная запись)

#### Принцип работы

Репозиторий не загружает сущность целиком для последующего сохранения, а выполняет UPDATE, чей WHERE включает все условия, необходимые для проверки инварианта. Метод возвращает `bool: True`, если строка была обновлена, `False` — если не найдена по условиям (конкурентная модификация или удаление).

Доменная сущность продолжает проверять инварианты на момент вызова метода (fail-fast), но результат проверки используется только для раннего выявления ошибок до обращения к БД, а не как гарантия отсутствия гонки.

```python
# Domain entity — проверка на момент вызова
def rotate_secret(self, new_secret: str, new_expires_at: datetime) -> None:
    self._ensure_valid()
    self.session_secret = new_secret
    self.expires_at = new_expires_at
    self._touch()

# Repository — атомарный условный UPDATE
async def rotate_secret(self, id: UUID, old_secret: str, new_secret: str, ...) -> bool:
    result = await conn.execute(
        update(sessions_table)
        .values(session_secret=new_secret, expires_at=new_expires_at)
        .where(
            sessions_table.c.id == id,
            sessions_table.c.session_secret == old_secret,
            sessions_table.c.expires_at >= datetime.now(UTC),
        )
    )
    return result.rowcount == 1
```

Такой подход уже реализован в `SqlAlchemySessionRepository.rotate_secret()` и `SqlAlchemyIdentityRepository.change_password_hash()`.

#### Преимущества

- **No retry required** — в случае гонки метод возвращает `False`, use case может немедленно ответить клиенту ошибкой без повторного чтения данных.
- **Отсутствие лишнего чтения** — UPDATE в БД — одна операция вместо select + update, что снижает время удержания блокировок строк (в InnoDB/PostgreSQL MVCC).
- **Совместимость с существующим Unit of Work** — не требует новых транзакций или уровня изоляции выше Read Committed.
- **Fail-fast в сущности** — проверка инварианта в Python срабатывает до SQL, отлавливая логические ошибки на раннем этапе (полезно при тестировании).

#### Недостатки

- **Инварианты дублируются** — условие `_ensure_valid()` в Python и эквивалентное условие в `WHERE` SQL должны быть согласованы. При рассинхронизации возможен сценарий, когда Python-проверка запрещает операцию, а SQL — разрешает (ложный отказ), или Python разрешает, а SQL блокирует (атипичная ошибка).
- **Нарушение чистоты DDD** — часть бизнес-правил неявно выражается в SQL инфраструктурного слоя, что противоречит строгой интерпретации Clean Architecture.
- **Только для простых инвариантов** — если условие зависит от данных, которые нельзя выразить в `WHERE` (например, агрегация по связанным таблицам), подход неприменим.

#### Ограничения

- Требует, чтобы все значимые для инварианта поля присутствовали в одной строке таблицы (или были выразимы через подзапрос в UPDATE).
- `bool`-сигнатура метода репозитория должна быть явной: `True` — обновлено успешно, `False` — не найдено/конфликт.

#### Влияние на архитектуру

Формально нарушает строгую Clean Architecture (правила конкурентного доступа проникают в инфраструктурный слой), но в рамках данного проекта это компромисс, уже принятый в коде. При этом Domain Layer остаётся независимым и проверяет инварианты самостоятельно; инфраструктура лишь страхует от гонки.

#### Влияние на тестируемость

Среднее: инвариант тестируется в двух местах — на уровне entity (Python) и на уровне репозитория (SQL). Требуются интеграционные тесты для проверки, что SQL-условие не расходится с Python-условием.

#### Влияние на сопровождаемость

Требует дисциплины: при изменении инварианта в entity необходимо синхронно изменить условие в репозитории. Рекомендуется документировать эту связь в docstring сущности (ссылкой на соответствующий метод репозитория).

#### Влияние на производительность

Позитивное: один round-trip к БД вместо двух, без retry-циклов.

#### Влияние на безопасность

Позитивное: гонка не приводит к неконсистентному состоянию — UPDATE либо применяется, либо нет, без промежуточных состояний.

### Вариант B: Оптимистичная блокировка через версионирование (version stamp)

#### Принцип работы

В агрегат и таблицу БД добавляется поле `version: int` (или `updated_at` как суррогат). При загрузке сущности репозиторий считывает версию; при сохранении — UPDATE с условием `WHERE id = ? AND version = ?`. Если `rowcount == 0` — кто-то изменил сущность, use case получает `ConcurrentModificationError`.

```python
class Identity(Identifiable[UUID], Auditable):
    def __init__(self, ..., version: int = 1) -> None:
        ...
        self._version = version

    @property
    def version(self) -> int:
        return self._version

# Repository
async def update(self, identity: Identity) -> None:
    result = await conn.execute(
        update(identities_table)
        .values(..., version=identity.version + 1)
        .where(
            identities_table.c.id == identity.id,
            identities_table.c.version == identity.version,
        )
    )
    if result.rowcount != 1:
        raise ConcurrentModificationError(identity.id)
```

#### Преимущества

- **Чистый DDD** — все инварианты проверяются в entity, репозиторий отвечает только за сохранение с проверкой версии.
- **Независимость от СУБД** — механизм реализуется на уровне приложения, а не SQL-диалекта.
- **Универсальность** — работает для любых инвариантов, включая проверки, требующие вычислений или доступа к другим агрегатам.
- **Понятный контракт** — `ConcurrentModificationError` однозначно трактуется: данные устарели, повторите операцию.

#### Недостатки

- **Дополнительное чтение** — даже для простого обновления одного поля требуется SELECT перед UPDATE.
- **Retry-цикл в use case** — при `ConcurrentModificationError` вызывающий код должен решить: повторить или сообщить клиенту об ошибке. Это либо усложняет use case, либо требует общей retry-инфраструктуры.
- **Расширение модели** — каждая сущность получает дополнительное поле `version` (или требует `updated_at` с достаточной точностью), что увеличивает объём данных и усложняет миграции.

#### Ограничения

- Эффективен только при низкой или средней частоте конфликтов. При high contention количество retry делает подход невыгодным.

#### Влияние на архитектуру

Полностью соответствует Clean Architecture: механизм конкурентного контроля реализован в репозитории (инфраструктурный слой) через проверку версии, домен не знает о нём.

#### Влияние на тестируемость

Высокая: `version` явно присутствует в модели, поведение при гонке детерминировано.

#### Влияние на сопровождаемость

Выше, чем у варианта A: нет дублирования инвариантов — при изменении логики достаточно изменить entity, версия обновляется автоматически при любом изменении.

#### Влияние на производительность

Отрицательное относительно варианта A: select перед каждым update, retry при конфликтах.

#### Влияние на безопасность

Позитивное: гонка приводит к явной ошибке, а не к "тихому" перезатиранию данных.

### Вариант C: SELECT ... FOR UPDATE (пессимистичная блокировка)

#### Принцип работы

Use case загружает сущность с блокировкой строки: `SELECT ... FOR UPDATE`. Транзакция удерживает блокировку до commit. Конкурентные запросы к той же строке ждут завершения первой транзакции.

```python
# Repository
async def get_by_id(self, id: UUID, lock: bool = False) -> Identity | None:
    stmt = select(identities_table).where(identities_table.c.id == id)
    if lock:
        stmt = stmt.with_for_update()
    ...
```

#### Преимущества

- **Максимальная гарантия** — никакая гонка невозможна, строки блокируются на уровне БД.
- **Простота использования** — use case не требует retry-логики.
- **Единая точка проверки** — инвариант проверяется только в entity, без дублирования.

#### Недостатки

- **Падение производительности** — блокировки удерживаются на всё время транзакции (чтение + бизнес-логика + запись), что снижает concurrent throughput.
- **Deadlock-риски** — при множественных блокировках в разном порядке возможно взаимоблокирование.
- **Необходим уровень Read Committed или выше** — требует, чтобы СУБД поддерживала блокировки строк (PostgreSQL, InnoDB — да, SQLite — нет).
- **Не масштабируется горизонтально** — при шардировании или read replicas `FOR UPDATE` не работает.

#### Ограничения

- Требует, чтобы транзакция была открыта на всём протяжении "чтение-модификация-запись", что противоречит подходу "транзакция минимальной длительности".
- Неприменим для high-contention сценариев.

#### Влияние на архитектуру

Допустимо: блокировка запрашивается в Infrastructure Layer, use case не знает о ней, но время жизни транзакции увеличивается.

#### Влияние на тестируемость

Низкое: тестирование поведения блокировок требует интеграционных тестов с параллельными соединениями.

#### Влияние на сопровождаемость

Негативное: неявная блокировка (спрятанная в репозитории) может быть неочевидна для разработчика, читающего use case. Если `lock=True` передаётся явно — сопровождаемость повышается, но сигнатура репозитория загромождается.

#### Влияние на производительность

Отрицательное при высокой конкурентности.

#### Влияние на безопасность

Максимальное: строгая сериализация доступа к строке.

### Вариант D: Domain events + асинхронная компенсация (eventual consistency)

#### Принцип работы

Use case не проверяет инварианты конкурентно, а принимает изменения и публикует доменное событие. Отдельный обработчик (saga) проверяет консистентность постфактум и при необходимости компенсирует изменения.

#### Преимущества

- **Максимальная пропускная способность** — нет блокировок, нет retry.
- **Асинхронная природа** хорошо сочетается с event-driven архитектурой.

#### Недостатки

- **Сложность на порядок выше** — требуется брокер сообщений, саги, компенсирующие транзакции.
- **Окна неконсистентности** — между записью и компенсацией данные находятся в противоречивом состоянии.
- **Не для всех операций** — `change_password`, `rotate_secret` должны быть строго консистентными; eventual consistency здесь неприемлема.

#### Ограничения

- В проекте отсутствует инфраструктура доменных событий и брокер сообщений.
- Требует ADR-расширения для доменных событий (см. ADR-0006, п. "Решение" — заготовка доменных событий отложена).

#### Влияние на архитектуру

Существенное: вводит новый механизм взаимодействия между агрегатами, требует инфраструктуры событий.

#### Влияние на тестируемость

Сложное: требуется тестирование саг и компенсирующих операций.

#### Влияние на сопровождаемость

Негативное для текущего этапа проекта: сложность не оправдана объёмом сценариев.

#### Влияние на производительность

Позитивное: нет синхронных блокировок.

#### Влияние на безопасность

Смешанное: окна неконсистентности допускаются архитектурно.

## Решение

1. **Основной подход (временно) — Вариант B: оптимистичная блокировка через version stamp.** Все новые методы-мутаторы репозиториев реализуются через проверку версии. Это обеспечивает чистоту DDD (инварианты только в entity) и даёт единообразный механизм для всех агрегатов.

2. **Точечно — Вариант C (SELECT FOR UPDATE).** Если для конкретного сценария окажется, что version stamp даёт слишком много retry (высокая частота конфликтов), допускается использование `SELECT FOR UPDATE` на уровне репозитория. Решение о переходе на C принимается на основании профилирования или наблюдения, а не заранее.

3. **Заложена возможность перехода на Вариант D (Domain Events + Saga).** При росте сложности и появлении инфраструктуры доменных событий допускается пересмотр данного решения в пользу eventual consistency для отдельных сценариев. Переход оформляется отдельным ADR.
