# ADR-0005: Поведение методов extend/rotate_secret для отозванной или истёкшей сессии

## Статус

Accepted

---

## Контекст

Доменная сущность `Session` содержит два независимых признака невалидности: `revoked_at` (принудительное завершение сессии) и `expires_at` (истечение срока действия). Помимо `revoke()`, сущность имеет методы `extend` (продление срока действия) и `rotate_secret` (замена секрета сессии с одновременным продлением — сценарий refresh).

На момент проектирования не было определено, должны ли `extend` и `rotate_secret` быть доступны для сессии, которая уже отозвана либо у которой истёк срок действия. Это прямой аналог решения, принятого в ADR-0004 для `User` и `is_active`: там же обсуждался риск того, что разные Use Case независимо друг от друга примут разные негласные допущения о поведении сущности в невалидном состоянии.

Цена ошибки здесь высокая: если `extend`/`rotate_secret` окажутся разрешены для уже отозванной сессии, это фактически создаёт способ восстановить сессию в обход logout — то есть дыру в логике завершения сессии. Требовалось явно зафиксировать инвариант, а не оставлять его на усмотрение Application Layer.

Необходимо определить, разрешены ли `extend` и `rotate_secret` для сессии с `is_valid() == False`.

---

## Требования

1. **Независимость домена** — правило должно быть реализовано исключительно средствами Entity, без обращения к Application/Infrastructure.
2. **Явность важнее магии** — ограничение должно быть выражено явной проверкой и явным исключением, а не побочным эффектом инфраструктуры (например, отказом при попытке записи истёкшей сессии в БД).
3. **Безопасность важнее производительности** — учитывая, что нарушение инварианта эквивалентно обходу отзыва сессии, при отсутствии явного use case приоритет отдаётся более строгому варианту.
4. **Документация является частью реализации** — принятое правило должно быть отражено в docstring сущности и оформлено ADR.
5. **Архитектурные решения принимает разработчик** — при наличии нескольких допустимых вариантов агент не выбирает поведение самостоятельно.
6. **Согласованность с ADR-0004** — по возможности поведение должно следовать уже принятому в проекте паттерну для аналогичной проблемы (`User`/`is_active`), чтобы не вводить два разных подхода к одному и тому же классу задач.
7. **Расширяемость** — правило должно допускать последующий пересмотр (например, grace period для истёкших, но не отозванных сессий) без изменения сигнатур методов.

---

## Рассматриваемые варианты

### Вариант A: Полный запрет `extend`/`rotate_secret` для невалидной сессии

#### Принцип работы

Оба метода в начале выполнения проверяют `is_valid()` (то есть одновременно `is_revoked() == False` и `is_expired() == False`) и возбуждают `SessionInvalidException`, если сессия отозвана или истекла. `revoke()` и `mark_used()` в эту проверку не входят: `revoke()` должен быть выполним в любом состоянии, а `mark_used()` не изменяет секрет или срок действия.

```python
# Пример (не реализация)
def extend(self, new_expires_at: datetime) -> None:
    self._ensure_valid()
    self.expires_at = new_expires_at
    self._touch()
```

#### Преимущества

- **Максимально строгий инвариант** — отозванную или истёкшую сессию невозможно "оживить" продлением срока или заменой секрета ни при каких условиях на уровне домена.
- **Единая точка проверки** — `_ensure_valid()` переиспользуется в обоих методах, что соответствует принципу "явность важнее магии".
- **Согласованность с ADR-0004** — в проекте уже есть прецедент того же класса решений (`User._ensure_active()`), что снижает когнитивную нагрузку при чтении кода: правило одинаково выглядит для обеих сущностей.
- **Устраняет конкретный риск обхода logout** — именно та угроза, из-за которой вопрос был поднят.

#### Недостатки

- **Потенциальная избыточная строгость** — при появлении легитимного refresh-сценария с grace period (например, разрешить `rotate_secret` в течение короткого времени после истечения `expires_at`, но до отзыва) потребуется точечно ослаблять инвариант.
- **`rotate_secret` для только что истёкшей сессии потребует создания новой `Session`** — это архитектурно нейтрально (просто другой Use Case: `issue()` вместо `rotate_secret()`), но требует, чтобы Application Layer уже сейчас это учитывал.

#### Ограничения

- Требует, чтобы `SessionInvalidException` был определён и доступен в Domain Layer.
- Любое расширение (например, grace period) потребует пересмотра ADR, а не тихого изменения кода.

#### Влияние на архитектуру

Полностью соответствует Clean Architecture: правило реализовано внутри Entity. Presentation Layer будет обязан замапить `SessionInvalidException` на HTTP-код (вне области текущего ADR).

#### Влияние на тестируемость

Позитивное: поведение детерминировано, покрывается компактным набором тестов (отозванная сессия / истёкшая сессия / валидная сессия — для каждого из двух методов).

#### Влияние на сопровождаемость

Высокая предсказуемость: разработчик, добавляющий новый метод изменения состояния сессии, обязан явно решить, относится ли он к "чувствительным" операциям (тогда вызывает `_ensure_valid()`) или нет (как `mark_used`/`revoke`).

#### Влияние на производительность

Незначительное — две проверки (`revoked_at is not None`, сравнение `datetime`) на вызов метода.

#### Влияние на безопасность

Позитивное: устраняет возможность продления или ротации секрета отозванной/истёкшей сессии, что напрямую закрывает риск, послуживший поводом для ADR.

### Вариант B: Полное разрешение, проверка — забота Application Layer

#### Принцип работы

`extend` и `rotate_secret` не проверяют `is_valid()`. Ответственность за проверку валидности сессии перед вызовом этих методов возлагается на конкретный Use Case (например, `RefreshSessionUseCase`).

```python
# Пример (не реализация)
def rotate_secret(self, new_session_secret: str, new_expires_at: datetime) -> None:
    self.session_secret = new_session_secret
    self.expires_at = new_expires_at
    self._touch()
```

#### Преимущества

- **Минимальный бойлерплейт в Entity**.
- **Гибкость** — любой сценарий (включая исключения из общего правила) реализуется без изменения домена.

#### Недостатки

- **Нарушает принцип "безопасность важнее производительности"** в трактовке приоритета строгости — инвариант, защищающий от обхода logout, отсутствует именно там, где цена ошибки максимальна.
- **Размывает ответственность Entity** — правило относится исключительно к собственному состоянию `Session` (валидна/невалидна), что по ARCHITECTURE.md — прямая зона ответственности Entity.
- **Несогласованность с ADR-0004** — в проекте возник бы прецедент, когда для одной сущности (`User`) аналогичный класс проблем решён строго на уровне Entity, а для другой (`Session`) — вынесен в Application Layer, без содержательной причины для различия.
- **Риск рассинхронизации** — при появлении нескольких Use Case, работающих с `Session` (login, refresh, admin-панель), каждый обязан самостоятельно помнить о проверке `is_valid()`.

#### Ограничения

- Требует дисциплины на уровне каждого Use Case, использующего `extend`/`rotate_secret`.

#### Влияние на архитектуру

Формально не нарушает Clean Architecture, но противоречит правилу "Entity отвечает исключительно за собственное состояние".

#### Влияние на тестируемость

Негативное: проверку пришлось бы тестировать в каждом Use Case отдельно, а не в одном месте на уровне Entity.

#### Влияние на сопровождаемость

Более низкая: при добавлении нового Use Case разработчик должен помнить о ручной проверке `is_valid()`, что не гарантировано архитектурно.

#### Влияние на производительность

Нейтральное.

#### Влияние на безопасность

Негативное относительно Варианта A: ошибка в одном Use Case открывает способ продлить или переподписать отозванную сессию.

### Вариант C: Дифференцированно — истёкшую, но не отозванную сессию разрешить (grace period)

#### Принцип работы

`rotate_secret` (и, возможно, `extend`) разрешён для сессии, у которой истёк `expires_at`, но которая не была явно отозвана (`revoked_at is None`) — классический паттерн refresh-token grace period. Для отозванной сессии оба метода запрещены безусловно.

```python
# Пример (не реализация)
def rotate_secret(self, new_session_secret: str, new_expires_at: datetime) -> None:
    if self.is_revoked():
        raise SessionInvalidException(self.id)
    # истечение expires_at не блокирует rotate_secret
    self.session_secret = new_session_secret
    self.expires_at = new_expires_at
    self._touch()
```

#### Преимущества

- **Ближе к распространённой практике refresh-токенов** — многие системы аутентификации допускают обновление в течение ограниченного окна после истечения.

#### Недостатки

- **Требует конкретного use case уже сейчас** — ни длительность grace period, ни сам сценарий не сформулированы как реальная потребность на данном этапе; вводить это заранее означало бы принимать архитектурное решение без достаточных оснований (см. также обсуждение Варианта C в ADR-0004 — тот же класс проблемы).
- **Усложняет модель** — вместо единого предиката `is_valid()` потребуется вводить дополнительное понятие "grace period", хранить его длительность (конфигурация домена или инфраструктуры?) и решать, применяется ли оно к `extend` так же, как к `rotate_secret`.
- **Нарушает принцип "архитектурные решения принимает разработчик"** — потребовало бы от агента самостоятельно определить длительность и границы grace period при отсутствии явного указания.

#### Ограничения

- Потребует дополнительного параметра (длительность grace period) и решения, где он конфигурируется.

#### Влияние на архитектуру

Не нарушает Clean Architecture, но вводит в Entity понятие, для которого пока нет обоснованных параметров.

#### Влияние на тестируемость

Среднее: требует дополнительных тестов на граничные случаи (сразу после истечения / после окончания grace period).

#### Влияние на сопровождаемость

Негативное на данном этапе: разработчику придётся сопровождать не самоочевидную границу между "истекла, но ещё можно" и "истекла окончательно".

#### Влияние на производительность

Незначительное.

#### Влияние на безопасность

Смешанное: расширяет окно, в течение которого скомпрометированный, но истёкший `session_secret` всё ещё может быть использован для ротации.

## Рекомендация

Рекомендуется использовать **Вариант A: полный запрет `extend`/`rotate_secret` для отозванной или истёкшей сессии**.

Обоснование:

1. **Соответствие принципу "безопасность важнее производительности"** — при отсутствии конкретного требования приоритет отдаётся варианту, устраняющему риск обхода `revoke()`.
2. **Согласованность с ADR-0004** — проект уже принял симметричное решение для `User`; применение того же паттерна к `Session` снижает архитектурную энтропию и упрощает аудит обеих сущностей по единому принципу.
3. **Отсутствие обосновывающего use case для Вариантов B и C** — ни делегирование проверки в Application Layer, ни grace period не подкреплены сформулированным сценарием на данном этапе проекта.
4. **Расширяемость** — при появлении конкретной потребности в grace period (Вариант C) решение может быть пересмотрено отдельным ADR, точечно ослабляющим инвариант, без изменения структуры сущности.

---

## Решение

Принят **Вариант A: полный запрет `extend`/`rotate_secret` для отозванной или истёкшей сессии**.

Оба метода проверяют `is_valid()` через `_ensure_valid()` и возбуждают `SessionInvalidException`, если сессия отозвана или истёк срок её действия. Методы `revoke()` и `mark_used()` данной проверке не подчиняются: `revoke()` выполним в любом состоянии сессии, `mark_used()` не изменяет чувствительные поля (`session_secret`, `expires_at`).

Варианты B и C отклонены по причинам, изложенным в разделе «Рекомендация».

Пересмотр решения в пользу grace period (Вариант C) допускается в будущем при появлении конкретного use case, но должен оформляться отдельным ADR с описанием сценария и обоснованием длительности окна, а не приниматься по умолчанию.
