# ADR-0006: Базовый класс Entity для доменных сущностей

## Статус

Accepted

---

## Контекст

В Domain Layer уже реализованы сущности `User` и `Session`, и, по словам разработчика, их количество будет расти. Обе существующие сущности независимо друг от друга дублируют один и тот же код:

- сравнение и хэширование по полю `id` (`__eq__`, `__hash__`);
- метод `_touch()`, обновляющий `updated_at`, с одинаковой сигнатурой (`at: datetime | None = None`), возникшей после исправления рассинхронизации временных меток в `Session.revoke()`.

При росте числа сущностей дублирование этого кода создаёт риск:

- рассинхронизации поведения между сущностями при точечных правках (например, если исправление, подобное найденному в `Session.revoke()`, будет внесено в одну сущность, но забыто в другой);
- увеличения объёма шаблонного кода, не относящегося к уникальной бизнес-логике конкретной сущности;
- нарушения принципа "явность важнее магии" не через магию, а через обратный эффект — многословный, повторяющийся код скрывает за собой то, что на самом деле является общим, универсальным для всех Entity поведением.

Одновременно требовалось не выйти за рамки уже принятых в проекте принципов: домен не должен приобретать неявное поведение (наследование не должно превращаться в скрытую магию), и агент не должен самостоятельно принимать решения там, где есть несколько допустимых вариантов (в частности — включать ли в базовый класс инфраструктуру для доменных событий, которые в проекте пока не реализованы, но заявлены в планах).

Необходимо определить, вводить ли общий базовый класс для доменных сущностей, и если да — какой объём поведения он должен инкапсулировать.

---

## Требования

1. **Независимость домена** — базовый класс должен располагаться в Domain Layer и не иметь зависимостей от внешних слоёв.
2. **Явность важнее магии** — наследование не должно скрывать инициализацию полей или подменять поведение неочевидным образом (никаких `__init_subclass__`, метаклассов, автоматической регистрации).
3. **Типобезопасность** — тип `id` не должен деградировать до `object`/`Any` при использовании базового класса, чтобы не терять проверки статического анализатора между сущностями с разными типами идентификаторов.
4. **Документация является частью реализации** — базовый класс и все переопределяемые сущностями методы должны быть задокументированы в стиле NumPy Doc.
5. **Архитектурные решения принимает разработчик** — объём базового класса (в частности, включение заготовки под доменные события) не определяется агентом самостоятельно при наличии альтернатив.
6. **Расширяемость** — структура должна допускать последующее добавление функциональности (например, доменных событий) отдельным ADR без разрушающих изменений в существующих сущностях.

---

## Рассматриваемые варианты

### Вариант A: Базовый класс `Entity(Generic[TId])` с идентичностью и `_touch`

#### Принцип работы

Вводится класс `Entity`, параметризованный типом идентификатора через `Generic[TId]`. Класс инкапсулирует инициализацию `id`/`updated_at`, сравнение по идентификатору (`__eq__`, `__hash__`) и метод `_touch()`. Конкретные сущности наследуются от `Entity[UUID]` (или иного типа идентификатора) и вызывают `super().__init__(id=..., updated_at=...)`.

```python
# Пример (не реализация)
class Entity(Generic[TId]):
    def __init__(self, id: TId, updated_at: datetime) -> None:
        self.id = id
        self.updated_at = updated_at

    def _touch(self, at: datetime | None = None) -> None:
        self.updated_at = at or datetime.utcnow()

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, self.__class__):
            return NotImplemented
        return self.id == other.id

    def __hash__(self) -> int:
        return hash(self.id)


class User(Entity[UUID]):
    def __init__(self, id: UUID, ..., updated_at: datetime, ...) -> None:
        super().__init__(id=id, updated_at=updated_at)
        ...
```

`__repr__` в базовый класс не выносится и остаётся специфичным для каждой сущности, так как разные сущности показывают разные содержательные поля (`is_active` у `User`, `revoked_at` у `Session`).

#### Преимущества

- **Устраняет дублирование** — идентичность и обновление временной метки описаны один раз.
- **Типобезопасность сохраняется** — `Generic[TId]` позволяет статическому анализатору видеть `User.id: UUID` и не допускать сравнения с идентификаторами другого типа, при этом в рантайме `Generic` не накладывает ограничений (Python не проверяет параметры типа во время выполнения), поэтому `isinstance`/`hash` работают без дополнительных условий.
- **Явность сохранена** — наследование единообразно и предсказуемо: `super().__init__()` вызывается явно в каждой сущности, никакой скрытой инициализации через метаклассы.
- **Согласуется с сериями ADR-0004/ADR-0005** — обе существующие сущности уже показали одинаковый паттерн проблем (инварианты состояния, рассинхронизация временных меток); общий базовый класс формализует то, что уже фактически является общим для Entity поведением.
- **Расширяемость** — при появлении доменных событий или дополнительных общих полей (например, `created_at`, если решат вынести и его) базовый класс можно расширить без изменения сигнатур существующих сущностей.

#### Недостатки

- **Ещё один уровень наследования** — при чтении кода конкретной сущности разработчику нужно заглянуть в `Entity`, чтобы увидеть полную картину доступных методов (`_touch`, `__eq__`, `__hash__`), а не только в саму сущность.
- **`created_at` не унифицирован** — по решению разработчика (см. раздел «Ответы на вопросы разработчику» ниже) в базовый класс выносится только `id`/`updated_at`, `created_at` остаётся полем каждой конкретной сущности, что создаёт асимметрию: одна временная метка общая, другая — нет.

#### Ограничения

- Требует, чтобы каждая новая сущность явно наследовалась от `Entity[TId]` и вызывала `super().__init__()` — дисциплина обеспечивается ревью кода, не языком.
- Не решает проблему дублирования `__repr__`, так как он сознательно оставлен специфичным.

#### Влияние на архитектуру

Полностью соответствует Clean Architecture: класс располагается в Domain Layer, не имеет внешних зависимостей, не содержит инфраструктурного кода.

#### Влияние на тестируемость

Позитивное: поведение идентичности и `_touch` тестируется один раз на уровне `Entity`, тесты конкретных сущностей могут сосредоточиться на их уникальной бизнес-логике.

#### Влияние на сопровождаемость

Позитивное: исправление, подобное найденному в `Session.revoke()` (согласование `updated_at` с другим полем через параметр `at`), теперь при необходимости меняется в одном месте и сразу доступно всем наследникам.

#### Влияние на производительность

Нейтральное — дополнительный уровень наследования не создаёт заметного оверхеда в CPython.

#### Влияние на безопасность

Нейтральное. Косвенно позитивное: единая реализация `__eq__`/`__hash__` снижает риск ошибки в сравнении сущностей (например, случайного сравнения по всем полям вместо `id`, что для Entity семантически некорректно).

### Вариант B: Отказаться от базового класса, оставить дублирование

#### Принцип работы

Каждая сущность продолжает независимо определять `__eq__`, `__hash__` и `_touch()`, как это сделано сейчас для `User` и `Session`.

#### Преимущества

- **Отсутствие дополнительной абстракции** — вся логика сущности видна в одном классе без необходимости смотреть на родителя.
- **Нулевой риск случайного наследования лишнего поведения**.

#### Недостатки

- **Дублирование растёт линейно с числом сущностей** — по словам разработчика, сущностей будет много, что делает этот вариант тем более обременительным.
- **Риск рассинхронизации** — уже был продемонстрирован на примере `Session.revoke()`: подобное исправление легко внести в одну сущность и забыть про остальные.
- **Противоречит "явность важнее магии" в широком смысле** — повторяющийся шаблонный код скрывает то, что фактически является общим инвариантом всех Entity, за видимостью независимости каждого класса.

#### Ограничения

Нет технических ограничений — вариант полностью соответствует статус-кво.

#### Влияние на архитектуру

Нейтральное — не вводит новых зависимостей, но не использует возможность формализовать общий контракт Entity.

#### Влияние на тестируемость

Негативное относительно Варианта A: идентичное поведение (`__eq__`, `__hash__`, `_touch`) приходится тестировать отдельно для каждой сущности.

#### Влияние на сопровождаемость

Негативное: при росте числа сущностей до "многих", как ожидает разработчик, объём дублируемого кода и риск рассинхронизации растут пропорционально.

#### Влияние на производительность

Нейтральное.

#### Влияние на безопасность

Нейтральное.

### Вариант C: Миксин только для `_touch`, без общей идентичности

#### Принцип работы

Вводится узкий миксин, отвечающий исключительно за `_touch()`, без вынесения `__eq__`/`__hash__`/инициализации `id`.

```python
# Пример (не реализация)
class TimestampedMixin:
    updated_at: datetime

    def _touch(self, at: datetime | None = None) -> None:
        self.updated_at = at or datetime.utcnow()
```

#### Преимущества

- **Минимальная площадь изменений** — устраняет дублирование только там, где оно было обнаружено (проблема с рассинхронизацией временных меток), не трогая остальное.

#### Недостатки

- **Не решает дублирование `__eq__`/`__hash__`** — а это дублирование не менее значимо: Entity как паттерн DDD определяется именно идентичностью по `id`, и вынесение только вспомогательного метода `_touch`, но не самой идентичности, разделяет концептуально единый контракт Entity на две независимые части без ясной причины.
- **Множественное наследование в перспективе** — если позже потребуется и идентичность, и `_touch`, придётся либо вводить второй миксin (`IdentifiableMixin`), либо всё равно прийти к варианту A, что означает два шага вместо одного.
- **Типобезопасность `id` не решается вовсе**, так как миксин её не касается — каждая сущность продолжает объявлять `__eq__`/`__hash__` самостоятельно, с риском, отмеченным в Варианте B.

#### Ограничения

- Не устраняет дублирование идентичности, только временных меток.

#### Влияние на архитектуру

Полностью соответствует Clean Architecture, как и Вариант A, но частично.

#### Влияние на тестируемость

Частично позитивное: `_touch` тестируется один раз, идентичность — по-прежнему в каждой сущности.

#### Влияние на сопровождаемость

Среднее: устраняет один из двух источников дублирования, оставляя второй нетронутым без содержательной причины для такого разделения.

#### Влияние на производительность

Нейтральное.

#### Влияние на безопасность

Нейтральное.

## Сравнительная таблица

| Критерий | Вариант A (базовый Entity) | Вариант B (без базового класса) | Вариант C (миксин только для _touch) |
|---|---|---|---|
| Устраняет дублирование идентичности | Да | Нет | Нет |
| Устраняет дублирование `_touch` | Да | Нет | Да |
| Типобезопасность `id` | Сохраняется (Generic[TId]) | Сохраняется, но раздельно в каждой сущности | Не затрагивается |

| Согласованность концепции Entity (идентичность + поведение) | Полная | Отсутствует единая формализация | Частичная, искусственно разделена |
| Готовность к росту количества сущностей | Высокая | Низкая | Средняя |

## Рекомендация

Рекомендуется использовать **Вариант A: базовый класс `Entity(Generic[TId])`**, включающий идентичность (`__eq__`, `__hash__`) и `_touch()`, без заготовки под доменные события.

Обоснование:

1. **Соответствие паттерну Entity из DDD** — идентичность по `id` является определяющим свойством Entity как концепции, и её формализация в общем базовом классе не выходит за рамки того, что архитектурно и так является общим для всех сущностей, а не специфичным решением конкретной из них.
2. **Типобезопасность не приносится в жертву** — параметризация через `Generic[TId]` позволяет сохранить статическую проверку типов идентификаторов, что было явным требованием разработчика.
3. **Соответствие "явность важнее магии"** — наследование остаётся явным (`super().__init__()` в каждой сущности), без метаклассов или скрытой регистрации.
4. **Готовность к росту** — при заявленном разработчиком росте числа сущностей вариант A линейно снижает объём дублируемого и потенциально рассинхронизируемого кода, что не обеспечивают ни Вариант B, ни Вариант C.
5. **Доменные события сознательно исключены из объёма** — по решению разработчика, заготовка под `_domain_events`/`pull_domain_events` не включается в данный ADR и будет рассмотрена отдельно, когда появится конкретная потребность в event-driven поведении.

---

## Решение

Принят **Вариант A: базовый класс `Entity(Generic[TId])`** с инициализацией `id`/`updated_at`, методом `_touch(at: datetime | None = None)` и реализацией `__eq__`/`__hash__` по идентификатору.

`__repr__` остаётся специфичным для каждой сущности и не выносится в базовый класс. `created_at` также остаётся полем каждой конкретной сущности, а не базового класса.

Заготовка под доменные события (`_domain_events`, `pull_domain_events`) в `Entity` **не включается** и будет рассмотрена отдельным ADR при появлении конкретного use case, требующего event-driven поведения.

Существующие сущности `User` и `Session` мигрируют на наследование от `Entity[UUID]`, с удалением дублирующихся `__eq__`/`__hash__`/`_touch` в пользу унаследованных из базового класса.

Варианты B и C отклонены по причинам, изложенным в разделе «Рекомендация».
