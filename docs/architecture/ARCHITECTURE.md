# ARCHITECTURE.md

# Архитектура проекта Elyria

## Назначение документа

Данный документ определяет официальные архитектурные правила проекта.

Все архитектурные решения принимаются исходя из правил данного документа.

При возникновении противоречий между существующим кодом и данным документом приоритет имеет данный документ.

---

# Архитектурные аксиомы

Проект построен на основе **Clean Architecture** с использованием отдельных принципов **Domain-Driven Design (DDD)**.

При разработке новых компонентов необходимо руководствоваться следующими принципами.

## Независимость домена

Домен является центром всей системы.

Ни один компонент Domain Layer не может зависеть от компонентов внешних слоёв.

Любая зависимость должна быть направлена к домену.

---

## Явность важнее магии

Предпочтение всегда отдаётся явным архитектурным решениям.

Если существует выбор между более короткой, но менее очевидной реализацией, и более длинной, но полностью прозрачной — выбирается второй вариант.

---

## Безопасность важнее производительности

При конфликте требований безопасность всегда имеет более высокий приоритет.

Оптимизация производительности допускается только после сохранения всех требований безопасности.

---

## Документация является частью реализации

Компонент без документации считается незавершённым.

Документация не является дополнительной задачей.

---

## Архитектурные решения принимает разработчик

AI-агенты не принимают архитектурные решения самостоятельно.

При наличии нескольких допустимых вариантов агент обязан подготовить ADR и дождаться решения разработчика.

---

# Слои приложения

Проект состоит из следующих слоёв.

* Presentation
* Application
* Domain
* Infrastructure
* Composition

Порядок зависимостей является строгим.

Нарушение данного порядка не допускается.

---

# Domain Layer

## Назначение

Domain Layer содержит предметную область приложения.

Домен не знает о существовании:

* HTTP;
* SQLAlchemy;
* PostgreSQL;
* Redis;
* MinIO;
* JWT;
* Cookie;
* FastAPI;
* Pydantic;
* внешних библиотек.

---

## Domain Entity

Entity отвечает исключительно за собственное состояние.

Если бизнес-логика относится только к внутреннему состоянию сущности, она должна быть реализована внутри самой Entity.

---

## Domain Service

Domain Service используется только тогда, когда бизнес-операция:

* относится исключительно к предметной области;
* не принадлежит естественным образом одной Entity;
* может быть реализована исключительно посредством Domain Entities и Value Objects;
* не требует обращения к инфраструктуре.

Domain Service не работает:

* с базой данных;
* с сетью;
* с файловой системой;
* с Redis;
* с HTTP;
* с JWT;
* с внешними API.

---

## Value Objects

Каждый инвариант предметной области должен быть представлен отдельным Value Object.

Под инвариантом понимается неизменяемое правило предметной области, не зависящее от реализации инфраструктуры.

Value Object не должен содержать знаний о конкретных технологиях.

---

## Исключения

Домен сигнализирует об ошибках посредством исключений.

Использование HTTP-исключений, кодов ответа или иных инфраструктурных механизмов внутри Domain запрещено.

---

# Application Layer

## Назначение

Application Layer реализует пользовательские сценарии (Use Cases).

Use Case отвечает за выполнение одной бизнес-задачи.

---

## Ответственность Use Case

Use Case может:

* получать данные через порты;
* вызывать Domain Entities;
* вызывать Domain Services;
* координировать выполнение нескольких операций;
* работать с несколькими репозиториями;
* обращаться к нескольким портам.

Use Case не должен содержать инфраструктурную реализацию.

---

## Зависимости

Application Layer не знает о:

* SQLAlchemy;
* Redis;
* JWT;
* HTTP;
* FastAPI;
* PostgreSQL;
* MinIO.

Все внешние зависимости описываются посредством портов.

---

## Порты

Порт описывает бизнес-возможность, а не конкретную технологию.

Например:

* Cache;
* File Storage;
* Password Hash Service;
* Token Service.

Названия портов не должны содержать сведений о конкретной реализации.

---

# Infrastructure Layer

## Назначение

Infrastructure содержит реализации портов Application Layer.

Все внешние технологии располагаются исключительно в данном слое.

Например:

* PostgreSQL;
* Redis;
* MinIO;
* JWT;
* Argon2;
* файловая система;
* внешние API.

---

## SQLAlchemy

В проекте используется исключительно SQLAlchemy Core.

Описание схемы базы данных осуществляется посредством `Table(...)`.

Работа с базой данных осуществляется посредством SQLAlchemy Core Expression Language (`select`, `insert`, `update`, `delete`).

Использование ORM запрещено.

---

## Репозитории

Репозитории реализуют интерфейсы (порты), определённые в Application Layer.

Репозиторий не должен самостоятельно выполнять commit транзакции.

---

# Presentation Layer

## Назначение

Presentation Layer отвечает за взаимодействие с внешними клиентами.

В рамках HTTP-приложения данный слой содержит:

* FastAPI;
* HTTP DTO;
* маршруты;
* middleware;
* dependency;
* OpenAPI;
* обработчики исключений.

---

## Валидация

Валидация выполняется на каждой границе между слоями.

Каждый слой обязан повторно проверить собственные инварианты независимо от результатов проверки предыдущего слоя.

---

## HTTP

HTTP-коды принадлежат исключительно Presentation Layer.

Domain и Application не должны содержать сведений о статус-кодах HTTP.

---

## Pydantic

Все HTTP-схемы располагаются только в Presentation Layer.

Общие схемы располагаются в:

`src/presentation/http/schemas`

Версионированные схемы располагаются в:

`src/presentation/http/v{version}/schemas`

---

# Composition Layer

## Назначение

Composition Layer выполняет сборку приложения.

Он отвечает за создание объектов и связывание компонентов между собой.

---

## Особый статус

Composition Layer является самым сильным слоем приложения.

По этой причине помещать сущности в данный слой следует только после анализа возможности их размещения в слоях ниже.

Composition не является местом для произвольного кода.

---

## Ответственность

Composition Layer отвечает за:

* создание объектов;
* внедрение зависимостей;
* создание Use Cases;
* создание реализаций портов;
* создание репозиториев;
* конфигурацию приложения;
* запуск приложения.

---

# Работа с базой данных

Во всём проекте используется SQLAlchemy Core.

Все SQL-запросы строятся посредством SQLAlchemy Expression Language.

Репозитории не выполняют commit.

Использование ORM запрещено.

Архитектура управления транзакциями будет определена отдельным ADR.

---

# Документация

Все публичные компоненты обязаны иметь документацию в стиле NumPy Doc.

Правила оформления:

* документация пишется на русском языке;
* заголовки NumPy Doc остаются на английском языке;
* первая строка содержит краткое summary;
* далее располагается подробное описание;
* обязательны блоки Parameters, Returns/Yields, Raises (при необходимости), Notes (при необходимости).

---

# OpenAPI

Каждый endpoint обязан содержать:

* summary;
* description;
* response description;
* response model;
* status codes;
* request examples;
* response examples;
* примеры ошибок;
* примеры комбинированных ошибок.

OpenAPI считается частью реализации.

---

# Definition of Done

Задача считается завершённой только при выполнении всех условий:

* реализована необходимая функциональность;
* соблюдены архитектурные правила;
* написана документация;
* оформлен OpenAPI;
* отсутствуют нарушения Code Style в рамках изменяемого кода.

Если в ходе работы обнаружены проблемы вне области текущей задачи, они фиксируются в отчёте и не препятствуют признанию задачи завершённой.
