# AGENTS.md

## Назначение

Данный документ определяет обязательные правила работы AI-агентов над проектом.

Все требования являются обязательными. Если выполнение задачи противоречит хотя бы одному правилу данного документа, агент обязан остановиться и сформировать отчёт, а не принимать решение самостоятельно.

---

# Основные принципы

## 1. Разработчик принимает архитектурные решения

Агент является исполнителем, а не архитектором.

Запрещается самостоятельно принимать решения относительно:

- структуры проекта;
- структуры каталогов;
- структуры пакетов;
- именования директорий;
- именования файлов;
- публичных интерфейсов;
- архитектурных паттернов;
- зависимостей между слоями;
- разделения ответственности компонентов.

При наличии нескольких корректных вариантов агент обязан:

1. перечислить варианты;
2. описать достоинства каждого;
3. описать недостатки каждого;
4. сформулировать собственную рекомендацию;
5. остановить выполнение задачи до получения решения разработчика.

---

## 2. Любые архитектурные изменения требуют согласования

Без согласования запрещается:

- создавать новые директории;
- удалять директории;
- переносить файлы;
- переименовывать файлы;
- изменять структуру проекта;
- менять публичные API существующих компонентов;
- менять соглашения проекта.

---

## 3. Новые зависимости

Добавление любых библиотек, фреймворков или инструментов требует предварительного согласования.

До получения согласия агент обязан предложить возможные варианты и объяснить причины их использования.

---

## 4. Изменения базы данных

Любые изменения схемы базы данных требуют предварительного согласования.

Это относится к:

- новым таблицам;
- новым индексам;
- ограничениям;
- изменениям существующих таблиц;
- миграциям Alembic.

---

## 5. Максимальное переиспользование существующего кода

Перед написанием нового компонента агент обязан:

1. найти существующие реализации;
2. оценить возможность их повторного использования;
3. использовать существующие решения при отсутствии объективных причин для переписывания.

Если существующая реализация имеет существенные недостатки, агент должен предложить её переработку.

---

## 6. Во время архитектурного аудита агенту запрещается:

• придумывать имена новых классов;

• придумывать имена новых файлов;

• придумывать структуру каталогов;

• предлагать конкретные реализации;

• выбирать библиотеки;

• выбирать паттерны.

Во время аудита необходимо описывать исключительно потребности проекта.

Например:

✔ Требуется сервис хеширования паролей.

✘ Требуется PasslibPasswordHasher.

Или:

✔ Требуется сервис генерации JWT.

✘ Необходимо создать JoseTokenService.

---

# Анализ проекта

Перед выполнением задачи агент обязан проанализировать существующий код.

Минимальный объём анализа включает:

- архитектуру;
- структуру каталогов;
- зависимости;
- DI;
- модели данных;
- существующие сервисы;
- существующие интерфейсы;
- конфигурацию;
- middleware;
- обработчики ошибок;
- OpenAPI;
- соглашения проекта;
- стиль написания кода.

---

## Выявленные проблемы

При анализе агент обязан составить список:

- архитектурных нарушений;
- нарушений принципов Clean Architecture;
- нарушений SOLID;
- потенциальных ошибок;
- технического долга.

Каждая проблема должна иметь уровень критичности:

- Critical
- High
- Medium
- Low

Для каждой проблемы необходимо предложить возможное решение.

Если проблема не влияет на выполнение текущей задачи, она должна быть вынесена в отдельный раздел "Рекомендации".

---

# Документация

Отсутствие документации считается незавершённой задачей.

Все новые компоненты должны иметь документацию.

Используется исключительно стиль **NumPy Doc**.

Требования:

- документация на русском языке;
- заголовки NumPy Doc — на английском;
- полная документация всех публичных компонентов.

---

# OpenAPI

Каждый новый endpoint обязан содержать:

- summary;
- description;
- response_description;
- request examples;
- response examples;
- описание всех возможных кодов ответа.

Для каждого ответа необходимо привести реальные примеры.

Если endpoint может вернуть несколько вариантов ошибки одного статуса, необходимо предоставить примеры каждого варианта.

При наличии комбинированных ошибок также требуется отдельный пример.

---

# Code Style

Агент обязан придерживаться существующего стиля проекта.

Если существуют разные варианты оформления, необходимо использовать тот, который уже преобладает в проекте.

---

# TODO и FIXME

При анализе агент обязан перечислить найденные TODO и FIXME.

Если они влияют на текущую задачу — предложить варианты решения.

Если не влияют — вынести отдельным разделом отчёта.

---

# Изменения вне области задачи

Если агент обнаруживает возможность улучшения проекта, которая не относится к текущей задаче, он не должен самостоятельно выполнять изменение.

Необходимо:

- описать проблему;
- объяснить преимущества изменения;
- предложить оформить отдельную задачу.

---

# Формат отчёта

Если задача требует анализа, итоговый отчёт должен содержать следующие разделы.

1. Что найдено
2. Что уже реализовано
3. Что отсутствует
4. Что потребуется изменить
5. Архитектурные риски
6. Предлагаемый план реализации
7. Выявленные проблемы
8. Рекомендации
9. Вопросы разработчику

---

# Если возникает неоднозначность

При любой неоднозначности агент обязан остановиться.

Запрещается:

- угадывать намерения разработчика;
- принимать архитектурные решения самостоятельно;
- выбирать вариант без согласования.

Вместо этого агент должен подготовить сравнительный анализ возможных решений и дождаться решения разработчика.
